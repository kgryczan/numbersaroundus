After mastering categorical feature engineering, it‚Äôs time to focus on numerical data transformations‚Äîa crucial step that can make or break model performance. Many machine learning algorithms assume numerical features follow a specific distribution, and incorrect handling can lead to biased models, slow training, and poor generalization.

In this article, we‚Äôll explore:
‚úÖ Why scaling matters and when to use it (and when to avoid it).
‚úÖ Different types of transformations‚Äîlog, power, and polynomial features.

üìå Chapter 1: Why Scaling & Transformation Matter

1Ô∏è‚É£ The Importance of Scaling Numerical Features

Machine learning models rely on mathematical operations that are sensitive to the scale of features. Features with different ranges can negatively impact model performance in several ways:

üìå Unbalanced feature influence ‚Üí Some models (e.g., linear regression, SVMs) give higher importance to large-magnitude features.
üìå Slow convergence in gradient-based models ‚Üí Features with large values dominate updates, making training slower and less stable.
üìå Distance-based models struggle ‚Üí Algorithms like KNN and K-Means depend on distance calculations, and unscaled data skews these distances.

üîπ Example: Predicting House Prices
Imagine we have two numerical features:

  • square_footage (ranging from 500 to 5000)

  • num_bedrooms (ranging from 1 to 5)

Since square_footage has a much larger range, some models might assign it more weight than necessary, even if num_bedrooms is just as important. Scaling ensures all features contribute fairly.

2Ô∏è‚É£ When Scaling Is (and Isn‚Äôt) Necessary

‚úÖ Models that Require Scaling:
‚úî Linear Regression, Logistic Regression (weights are sensitive to magnitude).
‚úî KNN, K-Means, PCA (distance-based models).
‚úî SVMs, Neural Networks (gradient-based optimizers).

‚ùå Models That Handle Scaling Automatically:
‚úñ Tree-based models (Random Forest, XGBoost, LightGBM)‚ÄîTrees split on feature values, so scaling does not affect performance.
‚úñ Naive Bayes‚ÄîUses probabilities, not distance-based measures.

üìå Chapter 2: Standardization vs. Normalization

Scaling numerical data is an essential preprocessing step, but not all scaling methods are created equal. Two of the most common techniques‚Äîstandardization and normalization‚Äîserve different purposes and should be applied depending on the dataset and model.

But before we dive in, let‚Äôs talk about something unexpectedly funny in R‚Äôs recipes package:

  • ‚ÄúNormalization‚Äù (step_normalize()) actually means standardization (Z-score transformation).

  • ‚ÄúRange Scaling‚Äù (step_range()) actually performs normalization (Min-Max scaling).

  • So, if you thought these names were flipped, you‚Äôre not alone! üé≠

Let‚Äôs break it all down.

üîπ How It Works
Standardization centers the data around 0 and scales it to have a standard deviation of 1. The formula is:

After standardization, the values are centered around 0, which prevents features with larger magnitudes from dominating smaller ones.

‚úÖ When to Use Standardization
‚úî Linear models (e.g., logistic regression, linear regression) ‚Üí Keeps coefficients balanced.
‚úî PCA (Principal Component Analysis) ‚Üí Reduces variance bias from large-magnitude variables.
‚úî SVMs, Neural Networks ‚Üí Ensures faster and more stable training.

‚ùå When Not to Use Standardization

  • When features have a non-normal distribution‚Äîsince standardization assumes a bell-curve shape.

  • For tree-based models (Random Forest, XGBoost, LightGBM)‚Äîscaling does not impact tree-based splits.

üìå R: Standardizing Features Using recipes (Even Though It Says ‚ÄúNormalize‚Äù)

library(tidymodels)

recipe_standardized <- recipe(~ sqft, data = df) %>%
  step_normalize(sqft)  # Standardization (Z-score)

prepped_data <- prep(recipe_standardized) %>% bake(new_data = df)

üìå Python: Standardizing with StandardScaler in scikit-learn

from sklearn.preprocessing import StandardScaler

scaler = StandardScaler()
df['sqft_standardized'] = scaler.fit_transform(df[['sqft']])

After normalization, all values fit within the 0-1 range, making comparisons between variables easier.

‚úÖ When to Use Normalization
‚úî KNN, K-Means, Neural Networks ‚Üí Models relying on distance-based calculations benefit from a uniform scale.
‚úî Data with extreme outliers ‚Üí Prevents large values from dominating calculations.

‚ùå When Not to Use Normalization

  • If the dataset contains meaningful negative values‚Äîscaling to [0,1] may distort relationships.

  • If a normal distribution is required‚Äîuse standardization instead.

üìå R: Normalizing Features Using recipes (Even Though It Says ‚ÄúRange‚Äù)

recipe_normalized <- recipe(~ Income, data = df) %>%
  step_range(Income, min = 0, max = 1)  # Min-Max Scaling (Normalization)

prepped_data <- prep(recipe_normalized) %>% bake(new_data = df)

üìå Python: Normalizing with MinMaxScaler in scikit-learn

from sklearn.preprocessing import MinMaxScaler

scaler = MinMaxScaler()
df['income_normalized'] = scaler.fit_transform(df[['Income']])

‚úÖ By choosing the right scaling method, we ensure models perform optimally without unnecessary transformations.

üìå Funny Naming Recap (So You Don‚Äôt Get Confused Again!)

What We Expect What recipes Actually Calls It What It Actually Does
Standardization (Z-score) step_normalize() Centers data (mean = 0, std = 1)
Min-Max Scaling (Normalization) step_range() Rescales values between [0,1]

Yes, it‚Äôs confusing at first, but just remember:

  • step_normalize() normalizes the distribution (standardization).

  • step_range() scales into a fixed range (normalization).

At the end of the day, names don‚Äôt matter‚Äîchoosing the right technique does! üé≠

üìå Chapter 3: Transforming Skewed Data

Scaling is not always enough to prepare numerical data for machine learning. Many real-world datasets contain skewed distributions, where values are concentrated in a small range and a few extreme values (outliers) dominate the feature.

If we don‚Äôt handle skewed data properly, it can lead to:
‚ùå Poor model performance (linear models assume normally distributed features).
‚ùå Ineffective scaling (min-max scaling doesn‚Äôt fix skewness).
‚ùå Reduced interpretability (exponential relationships may be misrepresented).

In this chapter, we‚Äôll explore:
‚úÖ How to detect skewed data.
‚úÖ Log, Box-Cox, and Yeo-Johnson transformations.
‚úÖ Feature binning (discretization) to improve model interpretability.

1Ô∏è‚É£ Detecting Skewed Data

Before applying transformations, we need to check whether a feature is skewed.

üîπ Right-skewed (positive skew) ‚Üí Long tail on the right (e.g., income distribution).
üîπ Left-skewed (negative skew) ‚Üí Long tail on the left (e.g., age at retirement).

üìå R: Checking Skewness in a Feature

library(e1071)

# Calculate skewness
skewness(df$income)

üìå Python: Checking Skewness with scipy

from scipy.stats import skew

print(skew(df['income']))  # Positive = right-skewed, Negative = left-skewed

‚úÖ A skewness value above ¬±1 indicates a highly skewed feature that may need transformation.

2Ô∏è‚É£ Log Transformation

üîπ How It Works
Log transformation reduces right skewness by compressing large values while keeping small values distinct:

\(X' = \log(X+1)\)

(The +1 prevents issues with zero values.)

üîπ Best for: Right-skewed data (e.g., income, sales, house prices).
üîπ Not useful if data contains negative or zero values.

üìå R: Applying Log Transformation

df$income_log <- log(df$income + 1)  # Log-transform income

üìå Python: Applying Log Transformation

import numpy as np

df['income_log'] = np.log1p(df['income'])  # log(1 + x)

‚úÖ Log transformation is a simple way to make right-skewed features more normal.
‚ùå Avoid using it on features with negative values!

3Ô∏è‚É£ Box-Cox Transformation (Only for Positive Data)

üîπ How It Works
Box-Cox transformation adjusts skewed data dynamically using a parameter Œª:

\[\begin{cases} \frac{X^\lambda - 1}{\lambda}, & \lambda \neq 0 \\ \log(X), & \lambda = 0 \end{cases}\]

üîπ Best for: Data that is not strictly right-skewed and requires a flexible transformation.
üîπ Only works for positive values!

üìå R: Applying Box-Cox Transformation

library(MASS)

df$income_boxcox <- boxcox(df$income + 1, lambda = seq(-2, 2, by = 0.1))  # Find best lambda

üìå Python: Applying Box-Cox Transformation

from scipy.stats import boxcox

df['income_boxcox'], lambda_opt = boxcox(df['income'] + 1)  # Apply transformation

‚úÖ Box-Cox adapts to different types of skewness dynamically.
‚ùå Only works on strictly positive values!

4Ô∏è‚É£ Yeo-Johnson Transformation (Handles Negative & Zero Values)

üîπ How It Works
Yeo-Johnson is similar to Box-Cox but works on both positive and negative values, making it more flexible.

\[\begin{cases} \frac{(X + 1)^\lambda - 1}{\lambda}, & X \geq 0, \lambda \neq 0 \\ \frac{-(|X| + 1)^{2 - \lambda} - 1}{2 - \lambda}, & X < 0, \lambda \neq 2 \end{cases}\]

üîπ Best for: Right- or left-skewed data, including negative values.
üîπ More robust than Box-Cox, since it handles negatives!

üìå R: Applying Yeo-Johnson Transformation

df$income_yeojohnson <- bestNormalize::yeojohnson(df$income)$x.t

üìå Python: Applying Yeo-Johnson Transformation

from sklearn.preprocessing import PowerTransformer

yeojohnson = PowerTransformer(method='yeo-johnson')
df['income_yeojohnson'] = yeojohnson.fit_transform(df[['income']])

‚úÖ Yeo-Johnson is the best choice when data contains negative values.
‚ùå Slightly more computationally expensive than log or Box-Cox transformations.

5Ô∏è‚É£ Feature Binning (Discretization)

Sometimes, instead of transforming continuous data, it‚Äôs better to convert it into categories (bins).

üîπ Best for:
‚úî Features that don‚Äôt need fine-grained numeric precision.
‚úî Making data more interpretable (e.g., income brackets).

üìå R: Binning Income into Categories

df$income_bin <- cut(df$income, breaks = c(0, 30000, 70000, 150000), labels = c("Low", "Medium", "High"))

üìå Python: Binning Income Using pd.cut()

df['income_bin'] = pd.cut(df['income'], bins=[0, 30000, 70000, 150000], labels=["Low", "Medium", "High"])

‚úÖ Binning can improve model interpretability, especially in decision trees.
‚ùå May lose fine-grained numeric detail.

üìå Summary: Choosing the Right Transformation

Transformation Fixes Handles Negative Values? Best For
Log Transform Right-skewed data ‚ùå No Income, sales, house prices
Box-Cox Flexible skew correction ‚ùå No Normally distributed features that need transformation
Yeo-Johnson Both right & left skew ‚úÖ Yes Datasets with negatives & zeros
Feature Binning Converts numeric to categories ‚úÖ Yes Making features more interpretable

‚úÖ By selecting the right transformation, we can make skewed data more usable for models while preserving interpretability.

üìå Chapter 4: Best Practices & Common Pitfalls in Numerical Transformations

We‚Äôve explored scaling and transforming numerical features, but applying these techniques without a strategy can lead to poor model performance, data leakage, or unnecessary complexity.

This chapter will focus on:
‚úÖ When to apply transformations before vs. after scaling.
‚úÖ How to prevent data leakage when using transformations.
‚úÖ Avoiding overfitting when using feature binning.

1Ô∏è‚É£ Should You Transform Before or After Scaling?

One common question in preprocessing is: ‚ÄúShould I apply log transformations or Box-Cox before or after standardization/normalization?‚Äù

üîπ Always transform first, then scale.

Why?
üìå Log, Box-Cox, and Yeo-Johnson are meant to fix skewness‚Äîscaling first would distort their effect.
üìå Scaling should be the last step before feeding data into the model, ensuring all features are on the same scale.

üìå Correct Order of Preprocessing Steps

Step What It Does Example Transformation
1Ô∏è‚É£ Handle missing values Avoids issues in transformation Mean/median imputation
2Ô∏è‚É£ Apply transformations Fixes skewed distributions Log, Box-Cox, Yeo-Johnson
3Ô∏è‚É£ Scale features Ensures uniform range Standardization (Z-score) or Min-Max Scaling

‚úÖ This ensures transformations work correctly before values are scaled.

üìå R: Correct Order in a recipes Pipeline

recipe_pipeline <- recipe(~ income, data = df) %>%
  step_impute_median(income) %>%  # Handle missing values
  step_log(income, base = 10) %>%  # Fix skewness
  step_normalize(income)  # Standardize feature (mean 0, std 1)

prepped_data <- prep(recipe_pipeline) %>% bake(new_data = df)

üìå Python: Correct Order in scikit-learn Pipeline

from sklearn.pipeline import Pipeline
from sklearn.preprocessing import StandardScaler, FunctionTransformer
import numpy as np

pipeline = Pipeline([
    ('log_transform', FunctionTransformer(np.log1p)),  # Apply log transformation
    ('scaler', StandardScaler())  # Standardize feature
])

df['income_transformed'] = pipeline.fit_transform(df[['income']])

‚úÖ Applying transformations before scaling ensures proper feature engineering.

2Ô∏è‚É£ Preventing Data Leakage in Feature Transformations

Feature transformations should only be learned from the training set‚Äîif applied to the full dataset before splitting, they can leak information into the test set.

üö® Common leakage risks:
‚ùå Fitting scalers or transformers on the full dataset (instead of just training data).
‚ùå Binning continuous values using test set information (test data should only be transformed using bins learned from training).

üìå R: Preventing Leakage in recipes

recipe_pipeline <- recipe(~ income, data = df_train) %>%
  step_log(income, base = 10) %>%
  step_normalize(income)

prepped_train <- prep(recipe_pipeline) %>% bake(new_data = df_train)
prepped_test <- bake(recipe_pipeline, new_data = df_test)  # Apply transformations to test set

üìå Python: Preventing Leakage in scikit-learn

from sklearn.model_selection import train_test_split

# Split data
df_train, df_test = train_test_split(df, test_size=0.2, random_state=42)

# Fit transformer on training set only
pipeline.fit(df_train[['income']])

# Apply transformation separately to test set
df_train['income_transformed'] = pipeline.transform(df_train[['income']])
df_test['income_transformed'] = pipeline.transform(df_test[['income']])

‚úÖ Always fit scalers and transformers on the training set and apply them separately to the test set.

3Ô∏è‚É£ Avoiding Overfitting in Feature Binning

Binning (discretization) can improve model interpretability, but if bins are too small, models may overfit.

üö® Common mistakes in binning:
‚ùå Using too many bins ‚Üí Models memorize categories instead of learning patterns.
‚ùå Defining bins based on test data ‚Üí This introduces data leakage.
‚ùå Creating bins without checking distribution ‚Üí May result in uneven data distribution.

üìå Best Practices for Feature Binning

‚úî Use quantile-based binning instead of equal-width bins.
‚úî Ensure bins capture meaningful groups (e.g., income brackets).
‚úî Test different bin sizes to prevent overfitting.

üìå R: Creating Quantile-Based Bins

df$income_bins <- cut_number(df$income, n = 4)  # Creates 4 equal-sized bins

üìå Python: Creating Quantile-Based Bins

df['income_bins'] = pd.qcut(df['income'], q=4, labels=["Low", "Medium", "High", "Very High"])

üìå Summary: Best Practices & Pitfalls in Numerical Feature Transformations

‚úÖ Best Practices ‚ùå Pitfalls to Avoid
Transform before scaling Scaling before transformation can distort effects.
Fit transformers on the training set only Applying to full dataset causes data leakage.
Use quantile-based binning instead of equal-width bins Too many bins lead to overfitting.
Choose the right transformation for the data distribution Using log transformations on negative values breaks models.

‚úÖ By following these principles, numerical transformations can significantly improve model performance without introducing bias.

üìå Chapter 5: Conclusion & Next Steps

Numerical feature transformations play a critical role in ensuring machine learning models are trained on data that is well-scaled, properly distributed, and interpretable. Throughout this article, we‚Äôve explored the best techniques to transform numerical features, prevent issues like skewness and overfitting, and apply these techniques effectively in R and Python.

1Ô∏è‚É£ Key Takeaways

‚úî Scaling matters for certain models‚Äîlinear models, distance-based models, and neural networks benefit from standardization or normalization.
‚úî Not all scaling methods are the same‚ÄîStandardization (step_normalize() in R, StandardScaler() in Python) is different from Min-Max Normalization (step_range() in R, MinMaxScaler() in Python).
‚úî Skewed features should be transformed before scaling‚Äîusing log, Box-Cox, or Yeo-Johnson transformations.
‚úî Prevent data leakage when applying transformations‚Äîalways fit transformations on the training set and apply them separately to the test set.
‚úî Binning can improve interpretability but must be done carefully‚Äîtoo many bins can lead to overfitting and poor generalization.

‚úÖ By selecting the right transformation technique, we ensure that numerical data is properly prepared for machine learning models, improving performance and interpretability.
